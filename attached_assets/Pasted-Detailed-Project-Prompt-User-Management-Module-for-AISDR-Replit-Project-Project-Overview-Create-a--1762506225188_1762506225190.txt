Detailed Project Prompt: User Management Module for AISDR Replit Project

Project Overview
Create a comprehensive user management system for an existing AISDR application on Replit that allows multi-user access with granular permission controls, role-based feature access, and isolated user workspaces.

Core Requirements
1. User Management Features
User Creation & Authentication

Invitation System

Admin sends invitation links via email (no password creation by admin)
Invitation link contains unique token with expiry (24-48 hours)
New user clicks link and creates their own password
Option to resend invitation if expired
Track invitation status (pending, accepted, expired)


User Authentication

Secure login/logout functionality
Password encryption and secure storage (bcrypt/argon2)
Email verification on first login
Two-factor authentication (optional but recommended)


Password Management

Self-service password reset via email
Password strength requirements
Password history (prevent reuse of last N passwords)
Force password change on first login (optional)


User Profile Management

Users can update their profile information
Avatar/profile picture upload
Contact information
Notification preferences



Session Management

Concurrent Login Handling

Option 1: Allow multiple simultaneous sessions from different devices
Option 2: Limit to single active session (logout other sessions on new login)
Option 3: Configurable per user/role
Display active sessions to user with ability to terminate specific sessions


Session Security

Configurable session timeout (e.g., 30 minutes of inactivity)
"Remember Me" functionality with extended token (7-30 days)
Secure session storage (httpOnly, secure cookies)
Session refresh mechanism before expiry
Auto-logout on suspicious activity


Session Monitoring

Track session start time, last activity, IP address, device info
Admin can view all active sessions
Force logout capability for admin (security purposes)

Database Indexing Strategy (Performance)
Critical Indexes for Performance
sql-- User table indexes
CREATE INDEX idx_users_email ON Users(email);
CREATE INDEX idx_users_status ON Users(status);
CREATE INDEX idx_users_role_id ON Users(role_id);
CREATE INDEX idx_users_is_active ON Users(is_active);
CREATE INDEX idx_users_deleted_at ON Users(deleted_at);

-- Composite indexes for common queries
CREATE INDEX idx_users_status_role ON Users(status, role_id);
CREATE INDEX idx_users_active_email ON Users(is_active, email);

-- User data indexes (most critical for multi-user performance)
CREATE INDEX idx_user_data_user_id ON User_Data(user_id);
CREATE INDEX idx_user_data_user_created ON User_Data(user_id, created_at);
CREATE INDEX idx_user_data_user_active ON User_Data(user_id, deleted_at);

-- Session indexes for quick lookups
CREATE INDEX idx_sessions_user_id ON User_Sessions(user_id);
CREATE INDEX idx_sessions_token ON User_Sessions(token);
CREATE INDEX idx_sessions_active_expires ON User_Sessions(is_active, expires_at);

-- Audit log indexes
CREATE INDEX idx_audit_user_id ON Audit_Logs(user_id);
CREATE INDEX idx_audit_created ON Audit_Logs(created_at);
CREATE INDEX idx_audit_action ON Audit_Logs(action);

-- Invitation indexes
CREATE INDEX idx_invitations_email ON User_Invitations(email);
CREATE INDEX idx_invitations_token ON User_Invitations(token);
CREATE INDEX idx_invitations_status_expires ON User_Invitations(status, expires_at);

-- Settings indexes
CREATE INDEX idx_settings_user_key ON User_Settings(user_id, setting_key);
Performance Optimization Guidelines

Use connection pooling for database connections
Implement query result caching for frequently accessed data
Use pagination for large data sets (limit/offset or cursor-based)
Implement lazy loading for related data
Monitor slow queries and optimize with EXPLAIN
Archive old audit logs periodically
Use database partitioning for large tables (by user_id or date)
Implement read replicas for read-heavy operations

Security Considerations
Authentication Security

Implement middleware to check permissions on every request
Use JWT tokens or session-based authentication with secure storage
Hash passwords with bcrypt or argon2 (cost factor 12+)
Implement rate limiting for login attempts (e.g., 5 attempts per 15 minutes)
Account lockout after repeated failed attempts
CAPTCHA on login after failed attempts

Data Security

Prevent SQL injection (use parameterized queries/ORMs)
Prevent XSS attacks (sanitize inputs, use CSP headers)
Implement CSRF protection (tokens for state-changing operations)
Encrypt sensitive data at rest (database encryption)
Use HTTPS for all communications
Implement input validation and sanitization
