1) Scheduler Service — execute scheduled automations

Goal: make the scheduled automation actually run (not just stored).

Typical approach: use a robust job queue like BullMQ (Redis-based) or a lightweight scheduler like node-cron if you prefer no Redis. BullMQ gives reliability, retries, visibility.

Files / services

schedulerService.ts — enqueues automation runs when scheduled

automationWorker.ts — worker that executes automationRun jobs (fetches prospects and enrolls)

jobs/automationProcessor.ts — main processor logic

BullMQ example (recommended)
// schedulerService.ts
import { Queue } from 'bullmq';
const automationQueue = new Queue('automation', { connection: redisConnection });

export async function scheduleAutomation(automationRunId, runAt) {
  // schedule delayed job
  await automationQueue.add('runAutomation', { automationRunId }, { delay: new Date(runAt) - Date.now() });
}

// worker.ts
import { Worker } from 'bullmq';
const worker = new Worker('automation', async job => {
  const { automationRunId } = job.data;
  await automationExecutor.execute(automationRunId);
}, { connection: redisConnection });

2) Exclusion Enforcement — apply rules when importing/saving prospects

Goal: apply exclusion rules (skip contacted, unsubscribed, duplicates) during ingestion (Apollo or existing).

High-level approach

Evaluate exclusions before saving/enrolling.

Exclusion checks: alreadyContacted (emails present in emails/emailQueue), unsubscribed (in unsubscribes), duplicate (same primaryEmail for user), do_not_contact list.

Apply in both Apollo import and existing-prospects flow.

Pseudocode (import pipeline)
async function filterAndSaveProspects(candidateContacts, userId, exclusionConfig) {
  const filtered = [];
  // batch query to reduce DB calls
  const emails = candidateContacts.map(c => c.primaryEmail);
  const existingProspects = await db.prospects.findMany({ where: { userId, primaryEmail: { in: emails } } });
  const unsubscribes = await db.unsubscribes.findMany({ where: { email: { in: emails } } });
  const recentContactEmails = await db.emails.findMany({ where: { to: { in: emails }, userId, createdAt: { gte: someWindow } } });

  for (const c of candidateContacts) {
    if (exclusionConfig.skipUnsubscribed && unsubscribes.map(u => u.email).includes(c.primaryEmail)) continue;
    if (exclusionConfig.skipDuplicates && existingProspects.map(p => p.primaryEmail).includes(c.primaryEmail)) continue;
    if (exclusionConfig.skipContacted && recentContactEmails.map(e => e.to).includes(c.primaryEmail)) continue;
    filtered.push(c);
  }
  // Save filtered as prospects (ensure userId set)
  return saveProspectsBatch(filtered, userId);
}

3) Rate Limit Enforcement (daily limit + delay)

Goal: ensure UI limits reflect and are actually enforced during sending to avoid mailbox blacklisting.

Two parts:

Enforce per-mailbox or per-user daily limit (e.g., 200/day)

Enforce delay between emails (e.g., 5s between sends or jittered)

Approaches

A. Centralized sender worker (recommended): queue items in Redis/BullMQ and the worker pulls respecting rate limits per mailbox/user using token-bucket or leaky bucket algorithm.

B. DB row-locking: increment counters atomically in DB, but harder to scale.

Token bucket example (per mailbox)
// rateLimiter.ts
const buckets = new Map(); // mailboxId -> { tokens, lastRefill }
function allowSend(mailboxId, config) {
  refillTokens(mailboxId, config.ratePerSecond);
  if (buckets.get(mailboxId).tokens > 0) {
    buckets.get(mailboxId).tokens -= 1;
    return true;
  }
  return false;
}

Integration into emailQueueService.processPendingEmails

Modify worker to check allowSend(mailboxId) before sending; if not allowed, re-queue with small delay (or leave as pending).

for (const queueItem of pendingEmails) {
  const mailbox = selectMailbox(queueItem.userId);
  if (!rateLimiter.allowSend(mailbox.id, mailbox.rateConfig)) {
    // push job back or wait
    await requeueWithDelay(queueItem.id, mailbox.rateConfig.delayMs);
    continue;
  }
  // proceed with send
}

Daily counters

Maintain mailbox_daily_counts or stored fields email_mailboxes.daily_sent updated atomically (use DB UPDATE ... SET daily_sent = daily_sent + 1 WHERE id = ...).

Reset daily counts at midnight via a scheduled job (Cron) or store last_reset_at to compute derived counts instead of resetting.

Delay enforcement

Use per-mailbox minDelayBetweenEmailsMs. When mailbox used, set nextAvailableAt = Date.now() + delay in Redis; worker will only pick mailbox when Date.now() >= nextAvailableAt.

Edge cases & pitfalls

Timezones: scheduled times should be stored in UTC and displayed in user's timezone. Use explicit timezone conversions.

Large imports: batch saves and use background jobs to avoid long transactions.

Mail provider limits: Gmail/GSuite often enforce per-account/day limits; prefer user-level limits and warn users when approaching provider limits.

Exclusion stale data: maintain up-to-date unsubscribe lists (sync from mailbox webhooks if available).

Threading: message-id generation must be stable to match replies; store messageId back on queue item.
